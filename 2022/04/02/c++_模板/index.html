发表于 2022-04-02 更新于 2022-05-28 分类于 C++ 阅读次数：

函数模板

  • 函数模板：函数体本身不会被编译。在函数调用出会实例化出相应的模板函数。

  • 模板函数：在函数调用点，编译器用用户指定的类型，从原函数模版实例化出一份函数代码

  • 函数模板、模板的特例化、普通函数的重载

    • 模板特例化优于函数模板
    • 模板特例化和普通函数：优先调用普通函数，但如果显示<类型>了，就调用模板特例化。
    • 严格来讲不构成重载，重载是函数名相同，这几个东西函数名不同啊

  • 代码

  • 模板的非类型参数：必须是整数类型（整数或地址/引用都可以），必须是常量，只能使用，不能修改

问题 模板定义处和调用处

  • 模板代码不能在一个文件中定义，另一个文件中使用的。模板代码调用之前，一定要看到模板定义的时候（也可以说是，模板定义的地方（文件），一定要可以看到模板代码调用处。），这样的话，模板才能够进行正常的实例化。
  • 常用方案：所以模板代码一般放在头文件中，然后在源文件中直接include包含
  • 不常用方案：
    • 在函数模板定义文件中：template bool compare<类型>(类型 a，类型 b)

原因：

  • 函数模板本身的函数体不会被编译。
  • 模板函数：在函数调用点，编译器用用户指定的类型，从原函数模版实例化出一份函数代码。
  • 因为c++是分离式编译。会单独编译文件，然后再link。如果函数模板单独放入一个文件a，但调用处却在其他文件b（文件b有该函数模板的声明）。那么这个文件a在编译时是看不到函数调用处的。自然也就不会实例化出函数代码，在.o文件中自然也就不会有函数符号。而.o文件链接时，b文件中UND未定义的引用需要在外部.o文件去寻找相应定义处。然而，那个a文件对应的.o文件中并没有相应函数符号，也因此，找不到。所以，会在链接linked时报错。！

类模板

  • 模板名 + 类型参数列表 = 类名

  • 构造函数 析构函数 本来要求和类名相同，但是C++特殊规定不需要写参数列表

  • 所以 构造函数析构函数不用加，其他出现模板的地方都应该加上类型参数列表

  • 类模板 -》实例化 -》 模板类

  • 类模板的选择性实例化。并且不是一下子实例化出类中所有函数的代码。

    • 函数代码中调用了哪个函数，才生成相应T的函数。

  • memcpy浅拷贝

    • memcpy是内存的二进制格式拷贝，将一段内存空间中内容原封不动的拷贝到另外一段内存空间中。
    • 如果拷贝的是内置类型的元素，memcpy即高效又不会出错，但如果拷贝的是自定义类型元素，并且自定义类型元素中涉及到资源管理时，就会出错，因为memcpy的拷贝实际是浅拷贝。

顺序栈代码

空间配置器allocator

  • 空间配置器：

    • 负责内存的开辟、释放；对象的构造、析构
    • 将内存的开辟和对象的构造分离开
    • 将内存的释放和对象的析构分离开

  • 没用空间配置器

  • 使用空间配置器 allocator

    • 负责内存的开辟和释放，负责在指定内存上构造对象，以及析构指定内存上的对象

  • 好处：

    • 分离了内存开辟和对象的构造，分离了内存的释放和对象的析构；
    • 不必开辟多少内存就构造多少不必要的对象(T* p = new T[size])
    • 应当是：T *p = _allocator.allocate(sz); 开辟了sz大小的内存，并没有构造任何对象

迭代器

  • 提供一种统一的方式，来透明的遍历容器
  • 一般实现成容器的嵌套类型
  • for each也是通过迭代器实现的。
  • begin 返回容器底层首元素的迭代器位置
  • end 返回的是容器末尾元素的后继位置

迭代器失效

迭代器为什么失效

  • 当前容器调用erase方法后，当前位置到容器末尾元素的所有迭代器都失效了

  • 当前容器调用insert方法后，如果没扩容，当前位置到容器末尾元素的所有迭代器都失效了

  • 如果insert之后扩容了，那么就全部失效

  • 实现：将要失效的迭代器的pVec指向置为null，当需要再次调用这个迭代器时，会先判断pVec是否为null，为null，则抛出异常

    失效后如何解决

  • 对失效迭代器进行更新操作

  • erase删除当前迭代器指向元素，返回下一位置元素的迭代器。

  • insert:当前位置增加新元素，原元素依次向后移动。

  • 如何处理

  • 不同容器的迭代器是不能比较运算的。

  • push_back

    • 当扩容的时候会使得所有迭代器失效
    • 当正常的时候，会使得end()迭代器失效

    • iterator用于提供遍历vector功能

关于容器里放指针

delete

  • delete指针会发生什么。
    • 调用指针指向的对象的析构函数
    • 释放指针指向的内存
    • 并没有释放指针本身所占空间。
  • 指针（内置类型）的析构函数是什么？
    • 为空。

析构函数补充（内置类型何时释放）

  • 在一个构造函数中，成员的初始化是在函数体执行之前完成的。且按照他们在类中出现的顺序（定义的顺序）进行初始化（构造函数初始化值列表的前后位置官位不会影响实际的初始化顺序）。
  • 在一个构造函数中，首先执行函数体，然后销毁成员。成员按初始化的逆序销毁。
  • 在一个析构函数中，不存在类似构造函数钟初始化列表的东西来控制成员如何销毁。析构部分是隐式的。成员销毁时发生什么完全依赖于成员的类型。销毁类类型的成员需要执行成员自己的析构函数。内置类型没有析构函数，因此销毁内置类型成员什么也不需要。
  • 因此，当一个A类中，有B类对象b，那么在~A()中，不需要去调用b.~B()，因此A类中的成员销毁是隐式的，完全依赖于成员本身的类型。b的销毁会在A的析构函数执行结束之后，隐式进行，也即自动调用b的析构函数，并释放内存（这步大概吧）。同理A类中的其他成员如int a，则调用a的析构函数（空），然后释放内存。
  • 因此，如果一个A类中，存在 int * p = new B()，如果我们不再析构函数中，进行delete p，指望着隐式销毁p所指向的B的话，是不会成功的。析构函数结束后，会自动调用 指针p的析构函数（为空），然后释放指针p的内存。这样，p所指向的内存就泄露了。因此，必须函数体内delete p。

vector里面放指针

  • vector的析构函数不会delete容器内的元素，只会调用容器内元素本身的析构函数，具体可以看上面的vector代码。也就是说，不会delete存在容器里的指针，即，需要用户手动控制释放这些指针指向的内存。
    • vector的析构函数里只会调用存储元素的析构函数。而指针本身的析构函数是空。不会调用指针指向对象的析构函数。
  • 所以如果要在容器内存指针，
    • 要么存智能指针，如shared_ptr。
    • 要么自己手动在vector生命周期将结束时，delete这些指针。
